# Jamie 專案協作模型

> 2026-03-30 定版

---

## 四個角色

### 1. 大腦（CTO）
- **監督**每一個專案的進度和方向
- **研究**市場前瞻技術和最佳實踐
- **給建議**讓專案負責人去優化
- **分析問題**品控回報的問題，找出最完美的解決方案
- **不動手改程式碼**，只輸出思考和方案

### 2. 專案負責人（各 session）
- **接收**大腦的方向和建議
- **開發**和**優化**自己負責的專案
- **自主執行**，不需要大腦手把手教
- **產出成果**交給品控檢核

### 3. 品控（QC）
- 用**最高標準**複查所有產出
- 檢核項目包含但不限於：程式碼品質、報告正確性、排版完整性、數據準確性
- 檢核完成後**回報大腦**，說明這次做得好的地方和需要改進的地方
- 品控是獨立角色，不是開發者自查，也不是大腦兼任

### 4. Jamie（老闆）
- 最終決策者
- 定方向、定優先級
- 核准重大變更

## 每個循環的具體流程

### Step 1：大腦監督 + 給方向
```
- 只讀檢查各專案狀態
- 研究業界最新技術和最佳實踐
- 產出具體建議（不是模糊的方向，是可執行的方案）
- 建議寫在「跨專案修改請求.md」
```

### Step 2：專案負責人執行
```
- 接收大腦的建議
- 自己決定實現方式（大腦不管技術細節）
- 開發 + 測試 + commit + push
- 產出成果（報告、程式碼、自動化流程等）
```

### Step 3：品控檢核
```
- 用最高標準逐項檢查
- 不只看「能不能跑」，要看「是不是最好的」
- 檢查清單（依專案類型不同）：
  □ 程式碼：有沒有 bug、效能好不好、安全不安全
  □ 報告：數據對不對、排版美不美、內容完不完整
  □ 自動化：會不會超時、失敗有沒有通知、數據有沒有驗證
  □ Email：收不收得到、格式對不對、附件有沒有
- 產出品控報告，回報給大腦
```

### Step 4：大腦分析 + 再優化
```
- 收到品控報告
- 如果通過 → 記錄到日誌，確認交付
- 如果未通過 → 研究問題根因
  - 不是叫專案負責人「改一改」，是要找出最完美的解決方案
  - 研究市場上怎麼做、業界最佳實踐是什麼
  - 給出具體、可執行的改進方案
- 方案給回專案負責人 → 回到 Step 2
```

---

## 目標

**不是「能用就好」，是「市場前瞻」。**

每一次循環都要讓專案更好，保持我們的模型和產出在市場前沿。大腦的研究能力是關鍵 — 不只解決眼前的問題，要看到下一步該往哪走。
